# Agent Rules — India Runs Project

> **These rules are MANDATORY for every agent working on this codebase.**
> Read this file at the start of every session. Follow every rule without exception.

---

## Rule 1: CONTEXT.md — Shared Context

**File:** `CONTEXT.md` (project root)

**Purpose:** Persistent shared memory across sessions. Every agent must read this file first, and update it before finishing.

### When to Read
- **Always** read `CONTEXT.md` as the **first action** in any session, before touching any code.

### When to Update
- **Always** update `CONTEXT.md` at the **end** of your work, before finishing.
- Update it if you made any meaningful change (new module, bug fix, architecture decision, etc.).

### Size Limit
- **Maximum: 300 lines.** If the file exceeds 300 lines, you MUST compact it before committing.
- Compaction strategy:
  1. Remove completed tasks from "Current Work" → move to "Recent History" (keep last 5 items only)
  2. Collapse verbose descriptions into single-line summaries
  3. Remove any section that hasn't been updated in 3+ sessions
  4. Merge related items into bullet points
  5. Never delete the "Architecture Decisions" or "Known Issues" sections — only compress them

### What Goes in CONTEXT.md
- Project status (what's built, what's in progress)
- Current active tasks (what the next agent should pick up)
- Architecture decisions and rationale
- Known issues and blockers
- Key file locations and their purposes
- Environment setup status
- Dependencies and configuration notes

### Structure Template
```markdown
# Project Context — India Runs

> Last updated: [DATE] by [AGENT]
> ⚠️ Lines: [N]/300 — keep under limit!

## Current Status
[1-2 sentences: what's done, what's next]

## Active Tasks
- [ ] Task 1 (assigned: [date], status: in-progress)
- [ ] Task 2 (assigned: [date], status: pending)

## Architecture Decisions
- [Decision]: [Rationale]

## Key Files
| File | Purpose |
|------|---------|

## Known Issues
- [Issue]: [Workaround]

## Environment
- [Setup status]

## Recent History
- [What was done last session]
```

## Rule 2: TASK_LOG.md — Activity Log

**File:** `TASK_LOG.md` (project root)

**Purpose:** Append-only log of every meaningful action taken by any agent. Think of it as a git log but more human-readable.

### When to Read
- Read to understand what previous agents did, before starting work.

### When to Update
- **Always append** a new entry at the **end** of the file before finishing.
- One entry per session/agent run. Never edit previous entries.

### What Goes in TASK_LOG.md
- What was built/changed
- What tests were run and their results
- What issues were encountered and how they were resolved
- What decisions were made and why
- Files created/modified

### Entry Format
```markdown
## [DATE] — [AGENT TYPE] — [SESSION ID]

**Task:** [What was requested]
**Status:** [completed | partial | failed]

**Changes:**
- `path/to/file.py`: [What changed]
- `path/to/file2.py`: [What changed]

**Tests:**
- [Test results]

**Decisions:**
- [Any decisions made with rationale]

**Issues:**
- [Any problems encountered]

**Next Steps:**
- [What should be done next]
```

### Size Management
- TASK_LOG.md has **no hard limit** but keep entries concise (max 20 lines per entry).
- If the file grows beyond 500 lines, archive older entries to `TASK_LOG_ARCHIVE.md` and keep only the last 50 entries.

## Rule 3: Git Conventions

### Commit Message Format
```
<type>(<scope>): <short description>

<optional body>
```

### Types
| Type | When to Use | Example |
|------|------------|---------|
| `feat` | New feature or module | `feat(search): add hybrid search orchestrator` |
| `fix` | Bug fix | `fix(scorer): handle null experience years` |
| `refactor` | Code restructuring (no behavior change) | `refactor(agents): extract prompts to separate file` |
| `test` | Adding/updating tests | `test(search): add RRF fusion unit tests` |
| `docs` | Documentation only | `docs: update README with setup instructions` |
| `chore` | Build, config, CI, tooling | `chore: add docker-compose for PostgreSQL` |
| `style` | Formatting, whitespace, linting | `style: run ruff formatter on all files` |
| `perf` | Performance improvement | `perf(search): cache FAISS index between queries` |

### Scope
Use the module name: `search`, `matching`, `agents`, `api`, `ui`, `ingestion`, `language`, `rationale`, `fairness`, `data`, `core`, `tests`.

### Rules
1. **One logical change per commit.** Never commit multiple unrelated changes.
2. **Max 100 chars** in the subject line.
3. **Body explains WHY, not WHAT** (the diff shows what).
4. **Reference issue/task** if applicable: `feat(search): add BM25 index (ref: TASK-003)`.
5. **Never commit** secrets, API keys, large data files, or model weights.
6. **Always run** `ruff check src/` before committing.
7. **Commit after completing each logical unit of work** — don't batch everything at the end.

### Atomic Commit Checklist
Before committing, verify:
- [ ] The commit contains exactly one logical change
- [ ] All new files are included
- [ ] No debug prints or temporary code
- [ ] No secrets or credentials
- [ ] Code passes lint (`ruff check src/`)
- [ ] Commit message follows the format above
- [ ] TASK_LOG.md is updated
- [ ] CONTEXT.md is updated (if architecture changed)

## Rule 4: Session Startup Protocol

Every agent MUST do this at the start of every session:

```
1. Read .agent-rules.md (this file)
2. Read CONTEXT.md
3. Read TASK_LOG.md (last 3 entries only)
4. Understand current state before making any changes
```

## Rule 5: Session Shutdown Protocol

Every agent MUST do this at the end of every session:

```
1. Run ruff check src/ (lint check)
2. Update CONTEXT.md with current state
3. Append entry to TASK_LOG.md
4. Commit changes with meaningful message
5. Push if remote is configured
```

---

## Rule 6: Commit Workflow

For every logical change:

```bash
# 1. Check what changed
git status

# 2. Stage specific files (not everything)
git add src/search/hybrid.py src/search/tests/test_hybrid.py

# 3. Lint check
ruff check src/

# 4. Commit with message
git commit -m "feat(search): add RRF fusion to hybrid search orchestrator"

# 5. Update CONTEXT.md and TASK_LOG.md
# ... edit files ...

# 6. Commit the context updates
git add CONTEXT.md TASK_LOG.md
git commit -m "docs: update context and task log after search implementation"
```

### What NOT to Commit
- `data/` directory (too large, generate locally)
- `data/indexes/` (built locally)
- `data/models/` (downloaded locally)
- `.env` (secrets)
- `__pycache__/` (should be in .gitignore)
- `*.pyc` files
- Model weight files (too large)
